Web3D 技術地圖（Three.js 之外＋IFC 直讀方案）

以「用途」分類：通用 3D 引擎、BIM/IFC、輕量展示、XR、GIS、資料視覺化、底層 WebGL、遊戲引擎匯出。 每張卡片都附上官方網站／官方文件或官方 GitHub。
提示：點「分類按鈕」或用搜尋框；可同時套用多個分類 IFC 直讀：通常透過 WebAssembly 解析（如 web-ifc）在瀏覽器端轉幾何＋語意 glTF/GLB：Web3D 最常見的傳輸格式（Khronos）

能力對照（快速判斷）

這張表只抓我們剛討論的重點：是否能直接讀 IFC、以及常見的「定位」。想要「工程協作」通常會選 BIM/IFC 類；想要「最快出效果」會選輕量 Viewer；想要「渲染特效」會偏通用引擎或 WebGPU。

類別 代表方案 IFC 直讀 典型定位
BIM / IFC IFC.js / web-ifc-viewer / web-ifc-three / xeokit 可（視方案與模型大小） 構件語意、查詢、剖切/量測/分類等工程互動
通用引擎 Three.js / Babylon.js / PlayCanvas 多為「需加載入器」 自訂互動系統、渲染特效、完整 UI/流程
輕量 Viewer <model-viewer> 否（只走 glTF/GLB） 最快展示、嵌入、分享、AR（但不做 IFC 語意）
GIS / 城市 CesiumJS 視資料管線 3D 地球、城市尺度、地理座標/大範圍資料
資料視覺化 deck.gl 偏資料層 大量資料圖層、分析、時間序列（常搭配地圖）
3JS

Three.js

通用 Web3D 框架：場景/相機/材質/載入器/互動，適合深度客製

通用引擎 客製化：高 IFC：需 Loader glTF/GLB：常用
  • 適合做「你想怎麼互動就怎麼互動」的系統：選取、聚焦、狀態上色、面板、事件流程
  • IFC 直讀通常要搭配 web-ifc 系列 Loader（見下方 BIM 卡）
官方網站 threejs.org 官方 GitHub
BJS

Babylon.js

完整引擎取向：渲染/材質/工具鏈齊全，並提供 WebGPU 支援文件

通用引擎 渲染表現：強 WebGPU
  • 偏「畫面＋引擎」：做展示級渲染、特效、完整互動流程很順
  • 想用 WebGPU 路線可參考官方支援說明
官方網站 官方文件：WebGPU
PC

PlayCanvas Engine

開源 WebGL 引擎＋協作 Editor 生態，常用於互動展示/配置器/廣告

通用引擎 交付速度：快 glTF：友善
  • 想要「引擎＋線上工具」協作時很方便
  • 也適合做產品配置器、互動展示、可分享的 3D 應用
官方網站 官方文件：Engine 官方 GitHub
IFC

IFC.js / That Open（IfcLoader）

BIM 工具箱取向：載入 IFC → 轉換/管理 → 建立工程互動（剖切、量測等）

BIM / IFC IFC 直讀：可 工程工具：多
  • 官方 IfcLoader 文件提供「把 IFC 轉成可用於引擎的資料結構」的工作流
  • 適合要做：構件選取、屬性面板、分類/隔離、剖切/量測等工程互動
官方文件：IfcLoader 官方文件入口
WIFC

web-ifc-three（Three.js 官方 IFCLoader）

在 Three.js 內「解析 IFC 並生成幾何」的官方 Loader 方向

BIM / IFC IFC 直讀：可 大型模型：看效能
  • 把 IFC 解析成 Three.js 幾何，並可查詢 IFC 資料
  • 如果你已經用 Three.js，這是最直覺的 IFC 直讀補強路線
官方 GitHub：web-ifc-three
WIV

web-ifc-viewer

在 web-ifc-three 上再加「工程工具」：量測、剖切、平面導覽等

BIM / IFC IFC 直讀：可 工具：量測/剖切
  • 定位是「快速搭一個 BIM viewer + 工具」：把工程互動做得更快
官方 GitHub：web-ifc-viewer
XKT

xeokit SDK

BIM/AEC Web viewer：主打高效能與大模型，常搭配預轉檔（XKT）加速

BIM / IFC 大模型：強 IFC：常用轉檔
  • 官方說明強調：可視覺化複雜 AEC 模型；常見流程是預先轉成 XKT 再在 Web 端快速載入
官方網站 xeokit.io 官方：轉檔/載入說明
<mv>

<model-viewer>

最省事的 3D/AR Web 元件：快速嵌入、快速分享（只支援 glTF/GLB）

輕量 Viewer 出效果：最快 IFC：不支援 glTF/GLB
  • 官方 FAQ 明確：只支援 glTF/GLB
  • 適合「展示/嵌入/AR」，不適合做 IFC 語意與工程工具鏈
官方網站 modelviewer.dev 官方 FAQ（格式支援） 官方 GitHub
AF

A-Frame

用「類 HTML」寫 3D/AR/VR；官方說明它是建在 three.js 之上的框架

XR / VR / AR 基於 three.js 上手：快
  • 適合快速做 WebXR 展示或教學 demo（宣告式寫法）
  • 需要更深度客製時，可回到 JS/DOM/three.js 層
官方網站 aframe.io 官方說明（架構） 官方文件
C3D

CesiumJS

3D 地球/地理視覺化：城市尺度、地理座標、巨量資料

GIS / 城市 地理精度：高 3D Globe
  • 適合把工程/城市資料放進「真實世界座標」裡檢視與疊圖
官方：CesiumJS 官方：學習資源 官方：Sandcastle
DK

deck.gl

GPU 驅動的大數據視覺化：Layer 組合、強調效能與精度（常搭配地圖）

資料視覺化 大數據：強 Layer
  • 偏「資料層」：交通、人流、熱區、時間序列，做成可交互的分析視覺化
官方網站 deck.gl 官方文件
RGL

regl

Functional WebGL 抽象：減少狀態管理負擔，適合 shader/渲染管線客製

底層 WebGL 控制力：高 工程成本：高
  • 適合想「更接近 WebGL」又不想手動管太多 state 的人
官方 GitHub 官方網站
OGL

OGL

極簡 WebGL Library：更少抽象、更靠近 WebGL，適合自寫 shader

底層 WebGL 輕量 功能少
  • 若你追求「最小抽象層」並喜歡自己控渲染細節，OGL 很合適
官方 GitHub 官方 Examples
U

Unity WebGL

用 Unity 直接 Build 到瀏覽器（產生 WebGL build 輸出，適合遊戲式互動）

引擎匯出到 Web 互動：強 載入成本：高
  • 適合重度互動或遊戲化體驗；但檔案量與載入成本通常比純 Web3D 框架高
官方文件：Build WebGL 官方網站
G

Godot（Exporting for the Web）

Godot 匯出到 HTML5：官方文件指出需瀏覽器支援 WebAssembly 與 WebGL 2.0

引擎匯出到 Web 開源 WebGL2/WASM
  • 適合用引擎做互動後再「發布到 Web」
官方文件：Exporting for Web 官方網站
GLB

glTF / GLB（Khronos）

Web3D 最常見的「傳輸格式」：強調體積與載入效率（官方稱為 3D Asset Delivery）

標準 / 規範 跨引擎通用 PBR
  • 幾乎所有 Web3D 工具鏈都會用到（含 Viewer 與引擎）
官方介紹（Khronos） 官方 Registry（規格） 官方 GitHub
GPU

WebGPU（W3C）

新一代瀏覽器 GPU API 規範：更貼近現代 GPU（相較 WebGL 更接近底層能力）

標準 / 規範 未來趨勢 W3C
  • 若你走 Babylon.js / 新渲染路線，WebGPU 會是重要基礎
官方規範（W3C TR） WGSL（著色語言規範）
